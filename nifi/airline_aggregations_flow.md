# NiFi Flow Guide

This document standardizes the NiFi flow from the original project into a simpler GitHub-ready version.

## Recommended Flow

```text
ListHDFS -> FetchParquet -> PublishKafka_2_6
                   \-> LogAttribute (failure/retry)
```

## HDFS source

- **Directory**: `/data/aggregations`
- **Purpose**: detect Parquet output files generated by Hive

## Processor setup

### 1. ListHDFS

Use `ListHDFS` to monitor `/data/aggregations`.

Suggested properties:
- Hadoop Configuration Resources:
  - `/home/bigdata/hadoop-3.3.6/etc/hadoop/core-site.xml`
  - `/home/bigdata/hadoop-3.3.6/etc/hadoop/hdfs-site.xml`
- Directory:
  - `/data/aggregations`

### 2. FetchParquet

Use `FetchParquet` to read the Parquet files detected by `ListHDFS`.

Attach a `JsonRecordSetWriter` so the output becomes JSON.

### 3. PublishKafka_2_6

Use `PublishKafka_2_6` to send the transformed records to Kafka.

Suggested properties:
- Topic Name: `aggregations`
- Bootstrap Servers: `localhost:9092`

### 4. LogAttribute

Route `failure` and `retry` paths to `LogAttribute` for debugging and traceability.

## Why this standardized flow?

The original project notes mention both `FetchHDFS + ConvertRecord` and screenshots showing `FetchParquet`.
For a cleaner portfolio version, this repository standardizes the approach around **FetchParquet** because it matches the Parquet delivery goal and makes the flow easier to explain.

## Validation

After starting the flow, validate the result with a Kafka consumer:

```bash
bin/kafka-console-consumer \
  --bootstrap-server localhost:9092 \
  --topic aggregations \
  --from-beginning
```
